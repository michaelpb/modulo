Modulo Tutorial, Part 1:
Your first component

Prerequisites Part 1 of this tutorial requires a prerequisite knowledge of HTML & CSS only. Parts 2 and onward require familiarity with JavaScript.

Other frameworks Modulo shares a common set of terminology with frameworks like React and Vue, and a similar syntax and component definition structure to Vue, Svelt, or Riot.js. Similar to Polymer (now litElement), it is based on the Web Components API that is standard in all modern browsers.

Including Modulo

Welcome to the Modulo tutorial. By following these short tutorials, you can learn how to use Modulo in your own projects.

The first step is to include the Modulo.js file. You can do this one of 2 ways:

  1. Using a script tag to directly link to the minified version on a CDN (Recommended)
  2. Downloading the Modulo.js and including it with a script tag, e.g. <script src="/js/Modulo.js">

Then, Modulo needs to be explicitly activated. Somewhere before the closing body tag of your HTML file, you should activate Modulo with the following code: <script>Modulo.defineAll();<script>

Defining your first component

Once you have included Modulo and activated it, you can define your first custom component. Right now, we'll do that all in one file, although soon we'll learn how to split your components into a separate file, which is the preferred way.

Why use "components"? Have you ever repeated the same HTML, CSS, and/or JavaScript while writing a website or or web app? Components let you store repeated code in one place. Once defined, components can be re-used everywhere, saving developer time and reducing mistakes. Furthermore, within teams, components libraries improve collaboration between developers.

Component definitions start with a "component" opening tag in the format of <component name="HelloWorld">. Modulo will scan a document for these tags, creating components for each one it encounters. Every component definition must specify a name, conventionally in UpperCamelCase (although technically component names are case insensitive).

Once defined, you can use a component by referring to it's name as though it were a plain HTML tag like <div>. Components can go anywhere that plain HTML tags can go, and can be styled with CSS the same way as well. Unlike plain HTML tags, you must use a dash (-) when referring to a component, such as by prefixing with an "x" as such: x-. This is a rule of the HTML5 Custom Components standard, which Modulo is based upon.

So, if a component is defined like <component name="HelloWorld">, then it can be referenced elsewhere like <x-HelloWorld></x-HelloWorld>.

Okay, enough talk, let's actually try this out!

Try it now

  1. Open up an HTML file in your preferred text editor. It's okay to start from scratch, or use an existing project.
  2. Open that HTML file in your preferred web browser as well.
  3. Copy & paste in the following code: 
    <!-- Define a custom component -->
<component name="HelloWorld">
    <template>
        Hello <strong>Modulo</strong> World!
    </template>
</component>


<!-- Reuse it anywhere, just like any other HTML tag -->
<x-HelloWorld></x-HelloWorld>

<p>In a P tag: <x-HelloWorld></x-HelloWorld></p>


<!-- Finally, include Modulo CDN & activate (at the bottom) -->
<script src="https://cdn.skypack.dev/modulo?min"></script>
<script>Modulo.defineAll();</script>
  4. Refresh the web browser to view the results.

The anatomy of a component definition

The central concept to Modulo is that of Component Parts. Because it is so central, saying Component Parts over and over got old, so it's typically shortened to CParts.

CParts: The Musical Think of CParts like the cast and crew of a musical. Each are assigned to a different task—some are more flashy and visible, others being stage-hands working behind the scenes—but they all work together to put on the same show!

All component definitions consist of some number of CParts. Thus, a component definition can be thought of as merely a collection of CPart definitions. When a component definition is later instantiated (i.e., when that component is used on the page), it in turn will instantiate (i.e. create an instance of) a CPart for each of the CPart definitions contained in the component definition. "Under the hood" of your component, each CPart will have a different role to contribute to the functionality of your component.

The first two CParts:

  1. Template - <template>

    Templates are where you put any arbitrary HTML code that you want your component to contain. For now, we'll just include some unchanging HTML. Later, we'll learn how to use "templating language" to control what HTML is produced in what circumstance.

  2. Adding CSS Instead of a Style CPart, you can always link a global CSS file the regular way to style your components. However, many developers prefer the simplicity of keeping everything in one place, e.g. the CSS with the component definition that it styles.

    Style - <style>

    Just like the <style> tag in HTML, the Style CPart allows us to write CSS for our component. CSS written here will be automatically prefixed so that it will only apply to your component and any HTML generated by the Template CPart.

Throughout Modulo documentation, there are these little embedded code editors. These allow you to immediately practice new concepts learned. For simplicity, the top and bottom of the component definition code is omitted. Instead, these sample code editors only focus on the CPart definitions within.
